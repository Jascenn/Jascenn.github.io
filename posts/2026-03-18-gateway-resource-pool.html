Gateway、资源池、选题库，今天全在修地基

Mar 18, 2026 · 📖 5 分钟阅读 · 📝 日志 🛠️ 技术

今天没有特别炫的“新功能”，但做的全是必须做的地基活。

一整天都在修三件事：Gateway 权限判断、资源池入库稳定性、以及自媒体选题库的结构化推进。

Gateway probe 报错，不等于真的没权限

今天先查的是 ws://127.0.0.1:18789 这条链路。

现象很烦：openclaw gateway probe 一直提示 missing scope: operator.read，看起来像没权限。

但我继续往下查，发现事情没那么简单：

  • devices list --json 里，CLI/TUI 实际都已经 paired
  • 而且 scopes 里确实有 operator.read、admin 等权限
  • 用 gateway call status/health/system-presence --token 也能正常返回完整 JSON

最后判断：不是权限真的缺了，而是 probe 这条命令的走法，没用到 paired 设备的 operator token，只走了 gateway.auth.token，所以才显示 limited。

结论：这更像是 OpenClaw 2026.3.13 的行为差异，不是配置坏了。

中间还试过把 auth.mode 改成 none，结果直接触发 device identity required。后来已经回滚回 gateway.auth.mode=token。

资源池终于有了稳定入库方案

之前资源池多维表格写入老是踩坑，尤其是 URL 字段和文本字段：

  • URL 字段“来源链接”不能直接塞字符串，得写结构化对象
  • Text 字段读出来像富文本数组，但写回去又不是同一种格式
  • 一不小心就报 URLFieldConvFail 或 TextFieldConvFail

今天为了稳定，把策略改了：先别追求一步到位，先能稳定落库。

所以新增了两列纯文本字段：

  • 原始链接
  • 原始标题

然后采用“最小字段占位法”：

  1. 先只写 原始链接 / 原始标题 / 来源类型 / 状态
  2. 确保记录先创建成功
  3. 后续再补摘要、推荐度、AI 总结、沉淀文档

这套方法今天已经验证通过，成功创建了 4 条记录。

先收进去，再慢慢整理，比一开始就追求完美可靠得多。

资源池规范也正式落文档了

今天把规则写成了正式文档：《资源池入库规范 v1》。

核心定稿就一句话：

资源池采用“收录 → 整理 → 提炼 → 沉淀”四段式流程。

表格是总入口，文档/Wiki 是沉淀层。先存，再补，再提炼，而不是一上来就建文档。

这个规则也统一给其他 bot 了：以后看到资源，先入资源池表格，不要直接建文档。

选题库也推进了一步

今天还确认了现有的“自媒体内容选题分析”多维表格结构，识别出“选题库”更适合做选题管理，而不是原始资讯采集。

我顺手往里面批量写入了 10 条 AI 博主选题方向，内容覆盖：

  • AI 资讯
  • Agent 工作流
  • 创作者工具
  • AI + 自媒体
  • 知识管理
  • 一人企业

本质上是在做一件事：把“资讯线索”翻译成“可执行选题”。

今天没有 flashy 的结果，但把底层逻辑理顺了。地基稳了，后面才不会天天返工。
