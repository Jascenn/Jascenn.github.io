OpenClaw 升级踩坑：两套安装源统一的正确姿势

Mar 21, 2026 📖 5 分钟 🐛 踩坑 🛠️ 技术

升级 OpenClaw 这件事，比我想象的麻烦一点。原因是机器上同时存在两套安装：一套通过 nvm 管理的 npm 全局包，一套通过 Homebrew 安装的。两套版本不一致，升级一套另一套还在跑旧版本。

问题的根源

macOS 上用 nvm 管理 Node.js 的时候，npm install -g 装的包只在当前 nvm 版本下生效。如果同时用 Homebrew 装了同一个工具，两者会互相干扰——PATH 里谁在前面，就用谁的版本。

这次的情况是：Homebrew 版本是旧的，nvm 版本是新的，但 shell 启动时 Homebrew 的路径在前，所以实际跑的是旧版本。

解决方案

最终的处理方式是统一到 nvm + npm 这一套，把 Homebrew 版本卸掉。升级命令：

npm install -g openclaw@latest

升级到 2026.3.13 之后，顺手把默认模型切到了 gpt-5.4，deepseek 暂时禁用（响应不稳定）。

gateway probe 的误报问题

升级之后发现 openclaw gateway probe 一直报 missing scope: operator.read，但实际上 devices list 显示权限是完整的。

排查了一圈，结论是：probe 命令走的是 gateway.auth.token 连接路径，不走 devices/operator token，所以显示 limited 是版本行为，不是真正的权限问题。实际功能不受影响。

这种"看起来报错但实际没问题"的情况挺烦人的。记录下来，下次看到这个报错不用再排查一遍。
